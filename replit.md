# Semantic Growth Index (SGI)

SGI turns every conversation into a measurable score: it tracks how the depth, variety and precision of a user's language and thinking evolve over time.

## Run & Operate

- `pnpm --filter @workspace/api-server run dev` — run the API server (port 5000)
- `pnpm --filter @workspace/sgi-app run dev` — run the web app
- `pnpm --filter @workspace/sgi-mobile run dev` — run the Expo mobile app
- `pnpm run typecheck` — full typecheck across all packages
- `pnpm run build` — typecheck + build all packages
- `pnpm --filter @workspace/api-spec run codegen` — regenerate API hooks and Zod schemas from the OpenAPI spec
- `pnpm --filter @workspace/db run push` — push DB schema changes (dev only)
- Required env: `DATABASE_URL` — Postgres connection string

## Stack

- pnpm workspaces, Node.js 24, TypeScript 5.9
- Web: React + Vite (dark/futuristic theme), wouter routing, react-i18next (IT/EN/ES)
- Mobile: Expo (React Native)
- API: Express 5
- DB: PostgreSQL + Drizzle ORM
- Auth: Clerk
- AI: OpenAI (gpt-4o-mini) via the Replit AI proxy
- Validation: Zod (`zod/v4`), `drizzle-zod`
- API codegen: Orval (from OpenAPI spec)
- Build: esbuild (CJS bundle)

## Product

- Core loop: the user converses; each conversation produces an SGI score across semantic metrics (Depth, Connectivity, Precision, Revision…) with growth tracked over time.
- Public surfaces: landing (`/`), How It Works (`/how-it-works`). Authenticated surfaces: dashboard, chat, settings.

## SGI landing & marketing principles

The landing's single job is to take a visitor from "I don't get it" to "I've seen my number" as fast as possible. Everything serves **showing and trying the SGI score**.

1. **Clarity first** — benefit-oriented headline (what you get, not how the algorithm works); the technical name comes second.
2. **Show the output** — a sample SGI score card / growth visual in the hero beats explaining it. The score is the visual hero.
3. **Don't dump the 11 metrics on the home** — show 2-3 emblematic ones in human language; the full list lives in How It Works.
4. **One dominant CTA, repeated** — low-friction, value-first ("Measure your SGI"). The "aha" must come before asking for payment.
5. **Aesthetic = precision instrument**, not generic colorful SaaS. Clean type, space, strong accent on the numbers, dark mode. Web and mobile must look like one product (shared design system).
6. **SEO/OG** — benefit-oriented meta titles/descriptions tied to real queries; curated Open Graph image for shares.

Domain `sgindex.work` is a **marketing** element: it belongs only on social/marketing cards, NOT on functional pages like How It Works (logo only there).

## User preferences

- User is Italian; communicate in Italian. Italian is the primary audience (default i18n language `it`).
- **Act decisively.** When direction is clear (e.g. a provided brief), execute the work completely in one pass — don't make the user re-ask or "force" the same request multiple times. Persist agreed direction (like the landing principles above) so it stays consistent across sessions.
- Keep IT/EN/ES copy parallel whenever landing/marketing copy changes.

## Vincoli permanenti

### Clerk (Replit-managed)
- L'istanza Clerk è **Replit-managed**: nessun accesso diretto a dashboard.clerk.com.
- Le email transazionali (verifica email, reset password) **non sono personalizzabili** e restano in inglese — limite accettato, nessuna migrazione a Clerk esterno.
- Configurazione possibile solo via: Auth pane Replit, props `appearance`/`localization` su `ClerkProvider` nel codice, Publishing → Domains per il dominio mittente email.
- **Non proporre mai migrazione a Clerk esterno.**

### DNS sgindex.work
- I nameserver sono **Name.com** (`ns1cny.name.com` ecc.) — registrar esterno, non Replit.
- Record DNS (CNAME per Clerk, ecc.) vanno aggiunti dall'utente nel pannello Name.com, non da qui.
